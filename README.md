## **1. 로컬 llm 기반 ai 챗봇**

### **프로젝트 개요**

#### **1. 목적**
기본적인 고객 문의를 자동 처리할 수 있는 AI 챗봇 개발

#### **2. 팀 구성**
- **기획 1명**: 챗봇 플로우 설계, 주요 FAQ 정리, 사용자 피드백 반영, 챗봇 데이터 관리  
- **개발 1명(본인 역할)**:
  - LLM 모델 선정 및 서빙
  - RAG(Retrieval-Augmented Generation) 파이프라인 구축
  - 각종 알림 Slack 전송
  - Slack 및 Google Drive 스프레드시트를 활용한 LLM 서버 및 데이터 관리

#### **3. 개발 및 운영 기간**
- **개발 기간**: 3월 중순 ~ 4월 중순 (약 1개월)  
- **운영 기간**: 4월 중순 ~ 5월 말 (종합소득세 계산 및 신고 기간 운영)

#### **4. 기술 스택**
- **백엔드**: Python, FastAPI, NumPy, Transformers  
- **프론트엔드**: React, JavaScript  
- **데이터베이스**: MySQL 8.0, ChromaDB  
- **인프라**: Ubuntu, Docker

#### **5. 인프라 환경**
- **GPU**: RTX 3090 Ti 1대  
- **CPU**: Intel i9-12900K 1대

### **주요 이슈 및 해결 과정**

#### **1) 컴퓨팅 파워 한계**
##### **문제점**
- 완전 생성형 AI 기반 고객 응대 요구사항 충족 불가
- 개인 사용 시 응답 생성에 **6초 소요**, 동시 요청 시 최소 **10초 소요** → 서비스 불가 판정

##### **해결 방법**
- 응답 속도를 1초 내외로 줄이기 위해 **입력/출력 토큰 제한**
- **RAG 방식 적용**하여 검색된 다량의 데이터를 기반으로 최적 응답 선택
- LLM의 출력을 **2~3토큰 단위 코드 제한** 후 FAQ 검색 기반 응답 제공

##### **최적화 결과**
- 단건 응답 속도 **1.2초**, 피크 시 **1.6초** 달성

##### **기존 유사도 검색만 사용하지 않은 이유**
- 세금 도메인 특성상 적절한 전문 임베딩 모델 부재 → **유사도 검색 정확도 30%**
- **RAG 방식으로 다량의 검색 결과 중 LLM이 최적 응답을 선택** → **정확도 56%**
- 최적화 결과를 고려하여 **LLM 모델 사용 확정**

#### **2) 전문 인력 부족 문제**
##### **문제점**
- 프로젝트 진행 당시, LLM 및 Python 관련 지식이 부족

##### **해결 방법**
- 인터넷 및 서적을 활용하여 독학하며 개발 진행
- LLM 및 RAG 관련 최신 논문과 사례 분석

### **성과**
- 전체 고객 문의(약 24,000건) 중 **52% (12,500건)를 챗봇이 자동 처리**
- 유사한 FAQ 챗봇 도입 시 1만 3천 건 처리에 약 **700만 원 비용 소요** → 비용 절감 효과 확인
- 챗봇 도입 전 **고객 응답 시간 평균 56분**, 챗봇 도입 후 **고객 응답 시간 1분 내외**

### **향후 개선 방안**
- **도메인 특화 임베딩 모델 학습**: 세금 도메인에 특화된 모델을 학습하여 유사도 검색 성능 향상
- **양자 LoRA 학습 등 모델 최적화 기법 도입**: 챗봇의 정확도 상승 기대
- **대화 맥락 파악 기능 추가**: LangChain을 활용하여 장기 기억 파이프라인 구상

→ 6월 종소세 기간 종료 후 채널톡 도입으로 인해 챗봇 서비스 종료, 개선 방안 적용 불가

## **2. 카드사 스크래핑**

### **프로젝트 개요**

#### **1. 목적**
사용자가 등록한 인증서를 활용하여 카드사에 로그인하고, 카드 매입 및 매출 데이터를 스크래핑하여 자동으로 수집

#### **2. 팀 구성**
- **개발 1명(본인 역할)**
  - 인증서 로그인 구현
  - 인증서 등록 및 관리
  - 카드 사용 내역 스크래핑 로직 개발

#### **3. 개발 기간**
- **7월 중순 ~ 7월 말일 (2주간)**

#### **4. 기술 스택**
- **백엔드**: Python, FastAPI
- **스크래핑**: JavaScript
- **인프라**: 사내 스크래핑 서버

### **주요 이슈 및 해결 과정**

#### **1) 스크래핑 경험 부족**
##### **문제점**
- 공인 인증서 기반 로그인의 암호화 방식에 대한 사전 지식 부족

##### **해결 방법**
- 공인 인증서 암호화 방식 및 프로토콜 학습 후 적용
- 기존 사례 및 레퍼런스를 분석하여 구현

#### **2) 카드사별 로그인 및 인증 방식 차이**
##### **문제점**
- 각 카드사마다 로그인 방식과 보안 정책이 상이

##### **해결 방법**
- 개발자 도구 및 프록시 툴을 활용하여 각 카드사의 로그인 프로세스 분석
- 공통된 흐름을 찾아 재구현하고, 개별 카드사에 맞는 예외 처리 추가

### **성과**
- **5개 주요 카드사(신한, 삼성, 우리, 롯데, 현대) 스크래핑 성공**
- **카드 사용 내역을 자동으로 수집하여 장부 시스템 구축에 기여**
- **기존 수동 입력 방식 대비 데이터 수집 속도 및 정확도 향상**

## **3. 데이터 마트**

### **프로젝트 개요**

#### **1. 목적**
MSA(Microservices Architecture) 구축을 위한 공용 데이터 관리 및 동기화 시스템 개발

#### **2. 팀 구성**
- **개발 1명(본인 역할)**
  - 데이터 동기화 및 관리
  - 데이터 검증 로직 구현
  - 어드민(Admin) 시스템 구축

#### **3. 개발 기간**
- **10월 초순 ~ 12월 초순 (약 2개월)**

#### **4. 기술 스택**
- **백엔드**: Java 1.8, Tomcat
- **데이터베이스**: MariaDB
- **데이터 동기화**: 실시간 데이터 적용 로직 개발

### **성과**
- **각 독립 서버 간 공용 데이터 동기화 자동화 성공**
- **어드민에서 데이터 관리 가능하도록 설계하여 유지보수 효율성 증대**
- **JPA 없이 SQL 기반 유틸리티를 활용한 데이터 관리 최적화**

## **2. 자체 프레임워크 사용성 개선**

- 디버깅 툴 사용 가능하도록 개선
- 백엔드 팀 주니어 전원 공유하여 테스트 환경 개선

## **3. DRM 분석 및 연구**

### **사용 기술**
- **도구 및 언어**: 크롬 개발자 도구, Frida, JavaScript, dex2jar, Python

### **프로젝트 개요**
- PDF 및 동영상 DRM 기술 분석 및 적용 연구
- 웹 동영상 서비스의 Widevine L3 DRM 분석 및 해제 성공
- PDF DRM은 전용 프로그램에서만 해제 가능하여 보안이 취약한 안드로이드 앱을 디컴파일
- 난독화된 프로그램에서 함수 후킹을 통해 암호화 해제 후 파일 저장 방식으로 DRM 우회
